# 0011: Use explicitly configured subprocess adapters

- Status: Accepted
- Date: 2026-07-17

## Context

ADR 0010 requires concrete checkpoint adapters to evolve and deploy independently from the Session Manager core. Linking adapters in-process would make their dependencies part of the daemon, weaken crash isolation, and require rebuilding the core. A persistent adapter service would add supervision, socket discovery, authentication, and version-skew complexity before the first version needs long-lived communication.

The core must transfer opaque binary checkpoints exactly, distinguish a crash before external mutation from an unknown outcome after acceptance, and avoid accidentally discovering or invoking software installed on the user's desktop.

## Decision

Invoke version 1 adapters as one-operation-per-invocation subprocesses behind the typed generic adapter port.

Register adapters explicitly by stable ID and canonical absolute executable path in an owner-only registry. Do not scan directories, search `PATH`, load shared libraries, download plugins, or execute a shell command. Reject unsafe executable ownership or permissions and an executable whose reported ID differs from its registration.

Load one validated registry generation at daemon startup and never watch external edits. Keep the SHA-256 source ETag distinct from the canonical adapter-generation UUID that revises the accepted source/projection pair, including validated hello metadata and availability. Runtime changes use only an explicit atomic reload with both expected tokens. A reload that changes either the accepted source ETag or public projection creates a new generation and event; only an exact source-and-projection no-op preserves the generation. Reload may add independent adapters, but it cannot remove or change an adapter referenced by any pending lifecycle intent, accepted or indeterminate operation, or release obligation. Each spawned invocation remains pinned to its accepted registration generation. That generation does not pin executable bytes between invocations: every requested invocation revalidates and opens the current registered file. Hello results are cached only by exact opened-file identity together with the accepted adapter generation. Adapter ID, protocol version, display name, capabilities, and format ranges must be deterministic for an executable identity and accepted generation. An uncached identity/generation pair is validated by running `hello` and the requested method from duplicate descriptors of the same opened image; the method runs only if hello metadata exactly matches the accepted public projection, otherwise the adapter is unavailable until explicit reload. Transient method failure does not silently mutate public availability. Compatible upgrades use atomic path replacement and must preserve adapter identity, protocol behavior, and every unreleased operation record across status, re-export, and release; concurrent mutation of an already opened inode is unsupported and fails when detected.

Use a versioned bounded JSON Lines control protocol over standard input and output. The core creates and durably records an operation UUID before invocation. A lifecycle-mutating operation may return a permitted non-success final before acceptance and any external effect. Otherwise it durably registers that UUID, emits an `accepted` envelope before its first external effect, and persists its terminal result before attempting to emit the final outcome. Read-only inspection and idempotent release are control requests with one final response and no acceptance phase. Status is always queryable by the core operation UUID through a new subprocess invocation when the original transport becomes acceptance-unknown or indeterminate. An adapter may provide an additional opaque token, but recovery does not depend on receiving it.

Transfer checkpoint payloads through inherited file descriptors, never JSON or adapter-visible storage paths. The core creates and owns the files, verifies exact length and SHA-256, and gives restore/discard inputs read-only access. A completed capture is retained by the adapter and can be re-exported byte-for-byte through a fresh core-provided descriptor when recovery follows transport loss.

Use a minimal allowlisted environment plus explicitly registered non-secret environment names. Open and validate the registered executable without following symlinks, bind hello validation to that exact file identity together with the accepted adapter generation, and execute the requested method from that opened image through descriptor-based execution so path replacement cannot substitute a different or unvalidated inode. Apply monotonic elapsed deadlines and bounded standard error. The core may terminate a child only before it has delivered a complete valid mutating request. After complete handoff it sends neither `SIGTERM` nor `SIGKILL`, because durable acceptance may race with observation of the acceptance envelope; timeout without observed acceptance becomes acceptance-unknown. The core tracks and reaps the exact child, durably records that proof before relying on absent status, and uses status inspection rather than assuming failure or replaying the operation. Only the spawning daemon may create the proof, but later daemons may trust its durable record; an uncooperative or unproven child leaves the intent visibly recovery-required.

## Consequences

Adapters can be installed, upgraded, crashed, tested, and released independently without entering the core address space. The same executable boundary is straightforward to fake and exercise through a reusable conformance suite.

Starting a process for each request costs more than an in-process call or persistent connection, but checkpoint lifecycle operations are infrequent and isolation is more valuable in version 1.

Adapters maintain accepted and terminal operation records across invocations, restart, and compatible upgrade without age-based expiry. A terminal core commit creates a durable release obligation only for an operation known to have reached acceptance; from that commit onward the core stops querying status and invokes only idempotent release, clearing the obligation in a later durable commit after `released: true`. Releasing an already absent record succeeds, which safely covers a lost earlier release response without requiring a tombstone. While a lifecycle intent is still pending, absence before observed acceptance proves no adapter effect only after the spawning daemon has reaped the exact original child and durably committed that proof; later daemons may trust the record, but without it absence remains non-authoritative. Absence for a pending intent after recorded acceptance is a protocol violation. Adapters that need persistent platform state maintain it in their own platform or durable adapter-owned state. A later protocol may add a persistent transport, but it must remain compatible with the core's typed port and opaque payload rules rather than leaking platform types into the core.
